# QA review on v3 — 2026-04-27

QA's job: stop bugs from shipping. Real failure modes only.

v3 has had multiple refinement rounds — much that was caught in earlier QA passes is now folded in. This pass focuses on what's still missing or genuinely new since the v3 rewrite.

## Will ship a bug

### Q1. `User.token` as cookie value is a DB-leak vulnerability

`User#ensure_token` does:
```ruby
update_column(:token, SecureRandom.hex(32)) if token.blank?
```

The same value is written to `cookies.signed[:user_token]`. If the database is ever read by an attacker (SQL injection, backup leak, dev access misuse), they have **directly forgeable cookies for every anonymous User**. Sign-cookie protection only verifies the cookie wasn't tampered with in transit; it doesn't protect against an attacker who can read the DB and craft a valid signed cookie.

**Fix**: store the **hash** of the token in DB, never the plaintext. Cookie carries plaintext; DB stores the hash. On request, hash the cookie value and look up by hash.

```ruby
class User < ApplicationRecord
  # The DB column is `token_digest`; the cookie carries the plaintext.
  attr_accessor :token  # transient

  before_create :assign_token

  def self.find_by_cookie_token(plaintext)
    return nil if plaintext.blank?
    digest = Digest::SHA256.hexdigest(plaintext)
    find_by(token_digest: digest, email_address: nil)
  end

  private

  def assign_token
    self.token = SecureRandom.hex(32)
    self.token_digest = Digest::SHA256.hexdigest(token)
  end
end
```

Migration: rename `users.token` to `users.token_digest` (string). Update `Mobile::BaseController` to set the cookie from `user.token` (transient) at create time and look up by hash on subsequent requests.

This is the same pattern Rails uses internally for session tokens. **Standard practice; v3 missed it.**

### Q2. Anonymous User creation on every visit creates DB pressure under bot traffic

`Mobile::BaseController#ensure_user!` creates a `User` row on **every visit without an existing cookie**. With `User.create!` triggering an `after_create :create_initial_will` callback, every visit creates 2 DB rows minimum (User + Will). Plus any callbacks that downstream code adds.

Under bot traffic (scrapers, search-engine crawlers hitting the marketing-side `/mobile/intro`), this could be thousands of garbage rows per hour. Cleanup is hourly via `AbandonedAnonymousUserCleanupJob` but the writes still happen.

**Fix**: only create an anonymous User when the user takes a meaningful action — specifically, when they **POST** something or hit `/mobile/onboarding/*`. Pages like `/mobile/intro` (marketing-style splash) don't need a User row. Add a `requires_user!` filter only on the actions that actually need it:

```ruby
class Mobile::BaseController < ApplicationController
  helper_method :current_user

  before_action :touch_user_if_present
  # Note: NO before_action :ensure_user! at base level

  private

  def current_user
    @current_user ||= find_current_user
  end

  def find_current_user
    # ... same as before, returns nil if no User found
  end

  def requires_user!
    @current_user = current_user || create_anonymous_user!
  end

  def touch_user_if_present
    current_user&.touch_last_seen! if current_user&.anonymous?
  end
end
```

Then in `Mobile::OnboardingController` and `Mobile::AuthController`:
```ruby
class Mobile::OnboardingController < Mobile::BaseController
  before_action :requires_user!
  # ...
end
```

This means `/mobile/intro` and similar marketing pages don't create User rows. Only when the user actually engages does the User get created. Bot-resilient by design.

### Q3. Encryption-deterministic + Postgres unique index — verify this works

v3 declares:
```ruby
encrypts :email_address, deterministic: true
```

Plus the partial unique index:
```ruby
add_index :users, :email_address,
  unique: true,
  where: "email_address IS NOT NULL",
  name: "index_users_on_email_address_when_set"
```

AR encryption (deterministic mode) stores the ciphertext in the column. The unique index is on the ciphertext, not the plaintext. Two users with `email_address = "test@example.com"` would have the same ciphertext (deterministic = same plaintext → same ciphertext), so the unique index correctly catches duplicates.

**But**: case sensitivity. AR encryption doesn't auto-downcase before encrypting. Two writes — `Test@Example.com` and `test@example.com` — produce **different** ciphertexts under deterministic encryption (different plaintexts). The unique index doesn't catch this duplicate.

**Fix**: normalise email before encryption. Add a `before_save` callback:
```ruby
class User < ApplicationRecord
  encrypts :email_address, deterministic: true
  before_save :normalise_email_address

  private

  def normalise_email_address
    self.email_address = email_address.downcase.strip if email_address.present?
  end
end
```

Plus a model-level uniqueness validation as belt-and-braces:
```ruby
validates :email_address, uniqueness: { case_sensitive: false }, allow_nil: true
```

Verify the test in W2.12 covers this edge case ("creating two users with email differing only in case fails uniqueness").

### Q4. `times_divorced` / `times_widowed` allow negative numbers

Schema:
```ruby
t.integer :times_divorced, null: false, default: 0
t.integer :times_widowed, null: false, default: 0
```

No CHECK constraint or model validation. Submitting `times_divorced=-1` saves successfully.

**Fix**: add `validates :times_divorced, :times_widowed, numericality: { greater_than_or_equal_to: 0, only_integer: true }` on Person. Cheap. Plus a CHECK constraint at the DB level if we're already adding CHECKs (PRE_LAUNCH_TODO).

### Q5. `Mobile::AuthController#secure_account` validation contexts misalign

v3:
```ruby
if user.save(context: :register)
  # ...
end
```

But `save(context: :register)` runs validations only with `on: :register`. The standard validations (encrypt email, create_will callback, `ensure_token`) ALL run regardless. That's fine.

**However**: if the User row was already validated when created (anonymous), and we now call `user.save(context: :register)`, the `email_address` and `password` validations fire — but `password` is being assigned via `user.password = params[:password]` which goes through `has_secure_password`. Need to verify the v3 plan correctly sets the password attribute (sets `password_digest` via `password=`), not the `password_digest` directly.

```ruby
# Correct
user.password = params[:password]  # has_secure_password handles it

# Incorrect
user.password_digest = BCrypt::Password.create(params[:password])  # bypasses validation
```

v3 shows `user.password = params.require(:password)` — correct. Just verify the test catches mistaken raw-digest assignment.

### Q6. Migration ordering — drop-onboarding_sessions deploys before code refactor

v3's W2.1 #8 (`DropOnboardingSessionsTable`) drops the table. v3 says: "Run order: 1, 2, 3, 4, 5, 6, 7 first (additive), then **the code refactor (W2.5 onwards) deploys**, then 8 last (drops the now-unused table)."

But Rails migrations run as a batch on deploy. If we have all 8 migrations in one deploy, they run in order. The code refactor doesn't deploy "between" migrations; the entire deploy includes the code change + all migrations.

**Issue**: if migration 8 runs BEFORE migration 1-7 are applied (some DB stuck mid-deploy), the table drop happens on a schema where the new tables (people, parentages, etc.) don't exist yet. But that's only a concern if migrations run partial-then-stop, which Rails generally doesn't allow.

**Correct framing**: the deploy must be atomic — all 8 migrations + the code refactor land together. Acceptable. **But the v3 plan's wording suggests sequential deploys, which is wrong.** Clarify: it's one deploy with 8 migrations + code refactor in any commit order, but they must all reach production together.

**Fix**: rewrite the v3 ordering note: "All 8 migrations + W2.3 (User refactor) + W2.4–W2.7 (controller refactor) + W2.8–W2.11 (UX, jobs, tests) ship in **one atomic deploy**. Migration 8 is last in the migration sequence (Rails will run them in timestamp order); the code refactor is in the same commit batch."

### Q7. `User#first_incomplete_path` implementation is referenced but not shown

v3 says: "same logic as the old OnboardingSession#first_incomplete_path." But the agent implementing has to find the old code (which gets deleted in W2.1#8). If the deletion happens before the migration, the reference is gone.

**Fix**: include the actual implementation in v4. Specifically:

```ruby
def first_incomplete_path
  return mobile_onboarding_welcome_path unless welcome_complete?
  return mobile_onboarding_location_path unless location_complete?
  return mobile_onboarding_family_path unless family_complete?
  return mobile_onboarding_extended_family_path unless extended_family_complete?
  mobile_onboarding_wrap_up_path
end

def welcome_complete?
  will_maker_person.present? && will_maker_person.first_name.present? &&
    will_maker_person.last_name.present? && will_maker_person.date_of_birth.present?
end

def location_complete?
  will_maker_person&.country_of_residence.present? &&
    will_maker_person&.nationality.present? &&
    will_maker_person&.domiciled_in_uk.present? &&
    will_maker_person&.currently_resident_in_uk.present?
end

def family_complete?
  current_step.in?(%w[extended_family wrap_up]) || completed_at.present?
end

def extended_family_complete?
  current_step == "wrap_up" || completed_at.present?
end
```

Inline these in v4's W2.3. Reduces agent confusion + risk of drift.

### Q8. The Stimulus `login_warning_controller` only fires on click, not on form submit-via-keyboard

v3:
```javascript
confirmIfNeeded(event) {
  if (this.hasOnboardingDataValue) {
    const ok = window.confirm(...)
    if (!ok) event.preventDefault()
  }
}
```

This is bound to the `click` event on the link. But the user could:
- Press Enter while focused on the link (keyboard activation) — fires `click`, OK
- Use browser back-forward navigation to reach `/mobile/login` — bypasses the link entirely
- Have the link open in a new tab via Cmd+click — link follows; new tab loads `/mobile/login`; warning bypassed
- Manually type/bookmark `/mobile/login` — warning bypassed

**Fix**: protect at the destination too. `Mobile::SessionsController#new` checks if current_user has onboarding data and renders a confirmation page instead of the login form, if so. The confirmation page asks "Yes, log in" / "No, go back."

Or accept the bypass gaps as acceptable v1 risk (most users follow the link normally). Document.

## UX holes / failure modes

### Q9. Login-from-onboarding cascade destroys partner Person who is also a Person in the existing User's graph

If user has an anonymous draft with spouse "Sarah" entered, and they log into an existing account that also has a Sarah... the destroy cascade removes the anonymous Sarah. The existing account's Sarah is unaffected.

That's correct behavior. But the user might lose preferences they set on the anonymous draft's Sarah (e.g., updated phone number). The existing account's Sarah retains old data.

**Acceptable**: documented as "logging in discards anonymous data." Worth a UX note: after login, show "Welcome back. Your existing data is loaded." Don't hide the discard.

### Q10. Person-picker dropdown may show too many options

v3's W2.6.1 person-picker shows "every existing Person in the user's graph other than the will-maker themselves and the current spouse." But if the user has 5 children + 3 friends + 2 cousins + 4 colleagues + 6 distant relatives, the dropdown is 20+ items. Hard to find.

**Fix**: filter to plausible co-parents only. Likely candidates: people with `relationship_kind: "spouse"` (current spouse already excluded), `"partner"`, `"other"`, `"extended_family"`. Probably exclude "professional", "colleague", "friend".

```erb
<%= f.select [...],
  options_for_select(
    [["+ New person", "new"]] +
    @existing_people
      .where.not(relationship_kind: ["self", "spouse", "child", "professional", "colleague"])
      .map { |p| [["#{p.first_name} #{p.last_name}".strip, p.id]] },
    selected: "new"
  )
%>
```

Or accept the long list as v1 cost; Epic 4 person-merge feature handles cleanup.

## Process gaps

### Q11. No test verifies encryption is actually applied

v3 says "AR encryption declared on PII." But there's no test that:
1. Creates a Person with `first_name = "Alice"`
2. Queries the raw SQL: `User.connection.execute("SELECT first_name FROM people LIMIT 1")`
3. Asserts the result is NOT `"Alice"` — it's ciphertext

Without this, a misconfigured `encrypts` declaration ships and PII writes plaintext. Hard to detect after the fact.

**Add to v4**: encryption-applied test in the test suite.

```ruby
RSpec.describe "PII encryption at rest" do
  it "stores Person.first_name as ciphertext, never plaintext" do
    user = User.create!
    person = user.people.create!(relationship_kind: "self", first_name: "Alice")

    raw = ActiveRecord::Base.connection.exec_query(
      "SELECT first_name FROM people WHERE id = #{person.id}"
    ).first["first_name"]

    expect(raw).not_to eq "Alice"
    expect(raw.length).to be > 30  # ciphertext is much longer than plaintext
    expect(person.first_name).to eq "Alice"  # decryption works at the model layer
  end
end
```

### Q12. No test for the cookie cutover at registration

v3 says "session[:user_id] set; cookies.delete :user_token." A test should verify this cutover:

```ruby
it "swaps user_token cookie for session_id cookie at registration" do
  # Create anonymous User via /mobile/open
  get mobile_open_path
  expect(cookies[:user_token]).to be_present

  # Register via /mobile/auth/secure-account
  user_id = User.last.id
  post mobile_auth_secure_account_path, params: { email_address: "test@example.com", password: "securepass1234" }

  expect(cookies[:user_token]).to be_blank
  expect(session[:user_id]).to eq user_id
end
```

### Q13. No load test for `AbandonedAnonymousUserCleanupJob`

If 10,000 anonymous Users are created in a day (legitimate or bot traffic), the cleanup job processes them in batches. With cascade-destroy through People, Marriages, Parentages, Wills, etc., each deletion is multiple SQL operations. The job needs to be both correct AND fast.

**Acceptable**: load test deferred to PRE_LAUNCH_TODO. But document in v4 that the cleanup job's SLA matters once traffic ramps.

## Nits

### Q14. Will-from-day-one creates a Will for every visitor including bots

The `User.after_create :create_initial_will` callback creates a Will for every User, including anonymous Users that may never finish onboarding. Combined with Q2 (creating Users on every visit), the wills table accumulates throwaway rows.

If Q2 is fixed (only create User on meaningful action), Will-from-day-one becomes acceptable. If Q2 isn't fixed, both grow together.

**Recommend**: fix Q2 first. Then Will-from-day-one is fine.

### Q15. The `Sessions` table for registered Users — what's its lifecycle?

Standard Rails session creation puts a row in `sessions` (the existing `Session` model from current code). When user logs out, session is destroyed (`session[:user_id] = nil`). 

v3 doesn't show how the existing `Session` model is updated to handle the new flow. Probably no changes needed since it's standard Rails — but a sentence confirming would help.

### Q16. The `Mobile::BaseController` flow has no clear flow diagram

The order of `find_current_user`, `requires_user!`, `touch_user!` is non-obvious. A small flow diagram (or pseudocode comment) at the top of the controller would help future maintainers.

### Q17. The `cleanup_stale_cookie!` method destroys without confirmation

If a User's `last_seen_at` is more than 3 hours ago, `cleanup_stale_cookie!` destroys them and clears the cookie. The user comes back, expecting their progress, and finds nothing. UX dialogue could improve: explain that data was cleared. But the user has no UI surface to see this — they just hit a fresh page. Acceptable v1.
