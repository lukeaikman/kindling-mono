# Step 5 — QA review of v1 plan (2026-04-27)

QA's job: stop bugs from shipping. Not to add defensive code. Real failure modes only, separated by severity.

## Will ship a bug

### B1. The `relationship_kind: "self"` marker is fragile

Workstream 3.1 uses `relationship_kind: "self"` to identify the will-maker Person within an OnboardingSession. The unique constraint enforcing "one self per session" doesn't exist anywhere — a `find_or_create_by` race condition or a bug elsewhere can produce two will-maker Persons per OnboardingSession.

**Fix**: add a partial unique index in the `CreatePeople` migration:

```ruby
add_index :people, :onboarding_session_id,
  unique: true,
  where: "relationship_kind = 'self' AND onboarding_session_id IS NOT NULL",
  name: "index_people_on_session_will_maker"
```

Same need for `relationship_kind: "spouse"` (one spouse per session at most):

```ruby
add_index :people, :onboarding_session_id,
  unique: true,
  where: "relationship_kind = 'spouse' AND onboarding_session_id IS NOT NULL",
  name: "index_people_on_session_spouse"
```

### B2. `FamilyStepUpdate` allows orphan Marriage rows

In `apply_partner!`:

```ruby
when "single", "widowed"
  destroy_existing_partnership!  # destroys Marriage AND partner Person
```

But what if a child references the destroyed partner Person via `Parentage`? The Parentage row's FK violates if we delete the partner. Currently no cascade-delete is declared on `Parentage` from `Person`.

**Fix**: when destroying the partner (because user changed status to single), first nullify or destroy any Parentage rows that reference the partner:

```ruby
def destroy_existing_partnership!
  partner = @onboarding_session.partner_person
  return if partner.nil?

  Parentage.where(parent_person: partner).destroy_all
  Marriage.where(will_maker_person: @will_maker, partner_person: partner).destroy_all
  partner.destroy!
end
```

Plus add `dependent: :destroy` on Person → Parentage as parent and as child:

```ruby
class Person < ApplicationRecord
  has_many :parentages_as_parent, foreign_key: :parent_person_id, class_name: "Parentage", dependent: :destroy
  has_many :parentages_as_child, foreign_key: :child_person_id, class_name: "Parentage", dependent: :destroy
end
```

### B3. Co-parent rows leak when child is removed from form

User adds child Charlie + co-parent Bob (someone-else). User then removes Charlie from the form and submits. Currently:

- The `apply_children!` loop iterates over submitted children
- Charlie's record is no longer in the params
- Charlie's Person + Parentage rows remain in DB
- Bob (the co-parent Person) also remains

**Fix**: at the start of `apply_children!`, find all existing child Persons for this session and destroy any that aren't represented in the new params. Plus destroy associated co-parent Persons (those with `relationship_kind: "other"` and a Parentage row only to a destroyed child).

```ruby
def apply_children!
  if @params[:has_children] != "yes"
    @onboarding_session.people.where(relationship_kind: "child").destroy_all
    @onboarding_session.people.where(relationship_kind: "other").destroy_all
    return
  end

  submitted_positions = Array(@params[:children]).each_with_index.map { |_, i| i }
  @onboarding_session.people.where(relationship_kind: "child").where.not(position: submitted_positions).destroy_all

  # ... existing apply loop ...

  # Cleanup orphaned "someone else" co-parents
  orphaned_co_parents = @onboarding_session.people.where(relationship_kind: "other").includes(:parentages_as_parent).select { |p| p.parentages_as_parent.empty? }
  orphaned_co_parents.each(&:destroy)
end
```

Add `position` to Person if not already there. Currently the migration in workstream 1.1 doesn't have `position` — needs adding.

**Schema fix**: add `t.integer :position, null: false, default: 0` to the `CreatePeople` migration.

### B4. Cohabitation start date is required for legal claims, but the form makes it optional

UK Inheritance (Provision for Family and Dependants) Act 1975 — a cohabitee of 2+ years before death can claim from the estate. Without `started_at` on a cohabiting Marriage, we can't compute eligibility.

The plan's `family.html.erb` makes `started_at` part of the partner fields but conditional on `relationship_kind`. Currently the form will let the user submit a cohabiting Marriage with NULL `started_at` if validation isn't enforced server-side.

**Fix**: validate `started_at` presence on cohabiting marriages:

```ruby
class Marriage < ApplicationRecord
  validates :started_at, presence: true, if: -> { kind == "cohabiting" }
end
```

And update the test scenario in `happy_path_spec` to assert this on a cohabiting variant.

### B5. Encryption + the data migration in 3.4 is broken

Workstream 3.4 (`MigrateChildrenPayloadToRecords`) reads from `OnboardingSession.children_payload` (plaintext JSON) and writes to `Person.first_name` etc. After workstream 2 declares encryption on Person, the writes work — but only if encryption is configured at the time the migration runs.

Order matters: workstream 1 (schema) → workstream 2 (encryption setup) → workstream 3 (data migration). The plan's stated order has 3 in the middle but doesn't enforce 2 must precede 3.4.

**Fix**: rename workstream 2 to "Workstream 1.5 — AR encryption setup" and execute it before workstream 3.4. Also: the data migration must run inside a code release that has both the new schema (people table) AND the encryption decoration — single deploy unit.

### B6. The `started_at` field for partners isn't `Marriage.started_at` — it's nowhere

Looking at the `apply_partner!` method:

```ruby
Marriage.find_or_create_by!(will_maker_person: @will_maker, partner_person: partner) do |m|
  m.kind = ...
  m.phase = "active"
  m.started_at = @params.dig(:partner, :started_at)
end
```

`find_or_create_by!` only sets attributes inside the block when CREATING. If the Marriage row exists already (user edits the form mid-flow and resubmits), the `started_at` won't update.

**Fix**: split into find-then-update:

```ruby
marriage = Marriage.find_or_initialize_by(
  will_maker_person: @will_maker,
  partner_person: partner
)
marriage.assign_attributes(
  kind: marriage_kind_from(@params[:relationship_kind], @params[:marriage_kind]),
  phase: "active",
  started_at: @params.dig(:partner, :started_at)
)
marriage.save!
```

Same pattern needed in `apply_children!` for Parentage updates — `find_or_create_by` with block won't update existing rows.

## UX holes / failure modes

### U1. What happens when the user picks "widowed" as their relationship_status?

The plan says "destroy any existing Marriage row" + bumps `times_widowed`. But:

- We don't ask about the deceased spouse's name or relationship to children
- If existing children have a Parentage with the deceased ex-spouse Person, that row is deleted (per B2 fix)
- The user has no way to say "my deceased spouse Sarah was the biological mother of Charlie"

This is acceptable scope-wise (we said deceased-co-parent details aren't captured in onboarding), but **the UI must communicate this clearly**. When a user changes from "married" to "widowed," the form should prompt: "We'll record that you've been widowed. If your children had a Parentage relationship with your deceased spouse, that will be removed. You can re-add them later."

**Fix**: add an inline confirmation in the family step when relationship_kind changes to "widowed" with an existing Marriage. Stimulus controller can handle this client-side.

### U2. What happens on form-submit when validation fails?

The plan's `update_family` rescues `ActiveRecord::RecordInvalid` and renders `:family` with `status: :unprocessable_entity`. But:

- If `apply_partner!` fails after `apply_children!` partially succeeds, we have child Person rows created but no Marriage. The transaction rolls back, but if any callbacks fired (e.g. `after_save` triggers), they may have side effects.
- More likely: the user sees an error message but no clear indication of which field failed.

**Fix**: ensure all validation errors are surfaced to `@errors` and `@person` properly. Test: submit invalid params (e.g., missing partner first_name with relationship_kind=spouse) and assert form re-renders with the field-level error.

### U3. The "someone else" co-parent has no email/phone, so we can't invite them later

The plan creates a Person for the someone-else co-parent with just first_name + last_name + relationship_to_child. Email and phone are NULL.

When Epic 4 ships invitations and the user wants to invite Bob (the co-parent) to confirm guardianship, there's no email to send to. The user must manually supply it later.

This is acceptable — the v1 form doesn't ask for email of a co-parent (would lengthen the step further). Just **note in the implementation** that the someone-else Person row has incomplete contact info that must be filled in Epic 4 if/when the user wants to invite them.

## Process gaps

### P1. No test coverage for the `someone_else` co-parent flow

The Capybara happy-path test in workstream 6 tests "yes, with partner Bob, biological parent" only. It doesn't test the "someone else" branch.

**Fix**: add a second branch to the same test (or a separate test) that walks through "someone else" creation and verifies the Person row + Parentage row are correct.

### P2. No test for "user changes relationship_kind from married to single" mid-flow

The plan's `destroy_existing_partnership!` is invoked, but the integration test doesn't verify the cascade behaviour. A future bug here (e.g., dangling Parentage rows) would silently accumulate.

**Fix**: add an integration test (request spec, not Capybara) that:
1. Submits family step with married + 1 child + partner-as-co-parent
2. Submits family step again with single
3. Asserts no Marriage rows, no partner Person, no Parentage from partner to child

### P3. Validation context mismatch with shipped code

Existing Rails code uses `OnboardingSession` validation contexts (`:welcome_step`, `:family_step`). The plan moves to `Person` validation contexts. **What happens to existing in-flight OnboardingSession records when this deploys?** Specifically:

- Production has zero in-flight users (no users yet), so safe.
- TestFlight may have testers mid-onboarding — their existing OnboardingSession will have all the columns dropped (workstream 1.4) but no Person rows yet (no migration backfill from the new shape).

**Fix**: workstream 3.4 (data migration) must also create the will-maker Person from existing `OnboardingSession.first_name/last_name/dob` BEFORE workstream 1.4 drops those columns. The plan already mentions migrating children, but not the will-maker themselves. Critical addition.

## Nits

### N1. `relationship_kind: "self"` is a code-smell

It conflates "will-maker identity" with "relationship descriptor." Cleaner: a separate boolean column `is_will_maker` on Person. Then `relationship_kind` is purely descriptive (spouse / child / parent / friend / etc.) and `is_will_maker` answers the "is this the user themselves" question.

**Defer**: not blocking for v1. The marker works. Refactor when the second user of `relationship_kind` (Epic 4 "your people") shows whether the conflation hurts.

### N2. The plan estimates 6 days but the QA fixes add ~1.5 days

Realistic estimate: **7-8 days** of focused work after QA fixes. Worth knowing for scheduling.

### N3. The data migration (workstream 3.4) lacks rollback testing

The `down` migration is documented as "lossy" but no test asserts that running `rake db:rollback` produces a sensible state. If the rollback misfires and leaves half-migrated data, the next migration deploy fails.

**Defer**: rollback testing is launch-readiness. PRE_LAUNCH_TODO entry would be appropriate.

### N4. The Capybara test runs `js: true` but the project may not have a JS driver configured

The test uses `selenium_chrome_headless`. If the Rails app isn't already set up for system tests with JS (check `spec/rails_helper.rb` or `test/application_system_test_case.rb`), the first test run will fail with driver-not-found.

**Pre-flight check**: run `bin/rails system:tests:js` (or equivalent) in dev to verify chromedriver is installed and Rails system tests can run with JS. If not, add to the v1 plan: "configure Capybara + selenium-webdriver in `config/application.rb` or system_test_case." 30 mins.

### N5. The `started_at` label in the Marriage form is contextual

Plan says: render `"Wedding date"` for married, `"Civil partnership date"` for civil_partnership, `"When did you start living together?"` for cohabiting. Implementation needs Stimulus to update the label on relationship_kind change. **Verify this is in `family_form_controller.refreshConditionals`** — currently the plan's controller doesn't update labels.

**Fix**: add to `refreshConditionals`:

```javascript
const startedAtLabel = this.element.querySelector("[data-field='started_at-label']")
if (startedAtLabel) {
  startedAtLabel.textContent = {
    "spouse": "Wedding date",
    "civil_partnership": "Civil partnership date",
    "cohabiting": "When did you start living together?"
  }[relationshipKind] || "Started"
}
```
