## Elon's nine challenges, with Claude's read on each

### Q1. Family-step UX expansion — premature?

**Challenge**: native app and current Rails code don't capture per-child co-parent data. We're adding it now because it enables Epic 4+ business logic (default heir calc, guardian-appointment defaulting, step-children opt-in). But Epic 4 is months away. Risk: Epic 4 product decisions might shift the data we need.

**Claude's call**: **in scope, user already affirmed**. Asking these questions at onboarding time is contextually natural. Deferring would require awkward retroactive "please clarify your earlier answers" prompts.

### Q2. Children-as-records — required, or could extended JSON suffice?

**Challenge**: keeping `children_payload` JSON and just extending the schema to capture co-parent data per child would be a smaller change today. JSON migration to Person + Parentage rows happens at Epic 3 secure-account.

**Claude's call**: **records now**. The family-step UI is rendering against real DB-backed Person rows. JSON-as-form-binding for nested co-parent / "someone else" data creates a pseudo-Person-in-JSON abstraction that's worse than just having Person rows from day one. Cost difference: ~half a day extra now, saves a day later.

### Q3. AR encryption — earns its place in v1?

**Challenge**: encryption-at-rest doesn't help internal testers. It's a launch-readiness feature.

**Claude's call**: **in scope, minimal**. Cost is `encrypts :first_name, :last_name, :date_of_birth` etc. on Person and User — half a day total. Adding later means re-running data migrations against potentially real test data, which is more painful than declaring it now.

### Q4. Capybara end-to-end suite — minimum viable?

Three options:
- **A**: One Capybara test covering happy path (welcome → wrap-up). ~1 day.
- **B**: Per-step Capybara tests covering branches. ~3 days.
- **C**: Skip until specific gaps surface. 0 days.

**Claude's call**: **A**. Test infrastructure earns its place by being used. One test validates the plumbing works (driver, fixtures, helpers, Stimulus interactions). Second test is cheap once plumbing is validated. Don't pre-build a system test suite that might not match how regressions actually surface.

### Q5. Cascade-delete integration test — Epic 2 or PRE_LAUNCH?

**Challenge**: this is GDPR right-to-erasure work. Internal testers won't trigger it. Belongs in PRE_LAUNCH_TODO.

**Claude's call**: **move to PRE_LAUNCH_TODO**. Drop piece 2.9 from Epic 2 roadmap; add to PRE_LAUNCH_TODO with detail.

### Q6. `Person.relationship_kind` enum — needed for v1?

**Challenge**: required by the "someone else" case in the family-step expansion (third-party co-parent rows need a descriptor). Could be free-text for v1, formalized later.

**Claude's call**: **enum from day one**. Free text invites typos. Cost is one migration. Enum values: `spouse | partner | child | parent | sibling | extended_family | friend | professional | other` (per earlier ERD work). The "someone else" case typically gets `other` or `extended_family`.

### Q7. Per-child counters (`times_divorced`, `times_widowed`)

**Challenge**: nullable int counters vs simple booleans. The boolean was simpler.

**Claude's call**: **counters in scope, user already affirmed**. Future-proof for multiple-marriages-and-widowings cases. Default 0, NOT NULL. `times_widowed > 0` reads as "yes."

### Q8. OnboardingSession → User consolidation

**Already deferred to Epic 3.** No change.

### Q9. paper_trail / paranoia / postgres CHECK enums / postgres hardening / encryption-mode-audit

**All already deferred to PRE_LAUNCH_TODO.** No change.
