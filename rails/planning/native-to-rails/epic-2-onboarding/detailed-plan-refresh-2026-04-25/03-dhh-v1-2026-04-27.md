# Step 4 — DHH detailed plan v1 (2026-04-27)

> **Note to AI agent**: this plan is detailed by design. Where it specifies file paths, method names, migration column lists, tests, that's not because they're clever — it's because ambiguity is how you produce bugs in a long agent session. Follow the letter. Where the plan asks you to make a judgment call, make it and note what you chose in the commit message.
>
> **What's forbidden**: no service objects (`app/services/` is off limits); no form objects; no gems beyond Active Record encryption (already shipped) and `image_processing` (already in Gemfile); no premature extraction of helpers, concerns, or shared modules until at least three duplicates exist; no defensive code for cases that can't happen; no rescuing exceptions you don't have a specific recovery for.

## What this plan ships

Six interlocking workstreams, executed in this order to minimise rework:

1. **Schema migrations** — additive, backward-compatible
2. **AR encryption decoration** — declare encrypted columns
3. **Children-as-records refactor** — migrate `children_payload` JSON → `Person` + `Parentage` rows
4. **Family-step UX expansion** — per-child co-parent capture
5. **Family-form bug fix + NI** — small UI fixes
6. **Capybara happy-path test** — single end-to-end test

Done-when: full Epic 2 onboarding flow walks from `/mobile/intro` through `/mobile/onboarding/wrap-up` with proper `Person` and `Parentage` rows materialised, encrypted PII at rest, all existing tests passing, plus one new Capybara test green.

`FamilyStepUpdate` is a **Plain Old Ruby Object in `app/models/`**, not a service object in `app/services/`. It's a model in the Rails sense — it encapsulates the multi-row create/update logic for a step. (DHH-blessed: see Basecamp's `MessageRendition` pattern.)

```ruby
# app/models/family_step_update.rb
class FamilyStepUpdate
  def initialize(onboarding_session, params)
    @onboarding_session = onboarding_session
    @will_maker = onboarding_session.will_maker_person
    @params = params
  end

  def apply!
    apply_partner!
    apply_children!
  end

  private

  def apply_partner!
    case @params[:relationship_kind]
    when "spouse", "partner"
      partner = @onboarding_session.partner_person ||
                @onboarding_session.create_person!(relationship_kind: "spouse")
      partner.update!(@params.require(:partner).permit(:first_name, :last_name))
      Marriage.find_or_create_by!(will_maker_person: @will_maker, partner_person: partner) do |m|
        m.kind = marriage_kind_from(@params[:relationship_kind], @params[:marriage_kind])
        m.phase = "active"
        m.started_at = @params.dig(:partner, :started_at)
      end
    when "single", "widowed"
      destroy_existing_partnership!
      @will_maker.update!(times_widowed: @params[:times_widowed].to_i) if @params[:relationship_kind] == "widowed"
    end

    @will_maker.update!(times_divorced: @params[:times_divorced].to_i) if @params[:times_divorced]
  end

  def apply_children!
    return if @params[:has_children] != "yes"

    Array(@params[:children]).each_with_index do |child_params, idx|
      child_person = @onboarding_session.people.find_or_create_by!(
        relationship_kind: "child",
        position: idx
      )
      child_person.update!(child_params.permit(:first_name, :last_name, :date_of_birth, :disabled, :lacks_mental_capacity))

      Parentage.find_or_create_by!(parent_person: @will_maker, child_person: child_person) do |p|
        p.kind = child_params[:relationship_kind]  # biological | adoptive | step | foster
      end

      apply_co_parent!(child_person, child_params[:co_parent])
    end
  end

  def apply_co_parent!(child_person, co_parent_params)
    return if co_parent_params.blank?

    case co_parent_params[:type]
    when "partner"
      partner = @onboarding_session.partner_person
      Parentage.find_or_create_by!(parent_person: partner, child_person: child_person) do |p|
        p.kind = co_parent_params[:relationship_to_child]
      end
    when "someone_else"
      co_parent = @onboarding_session.people.create!(
        relationship_kind: "other",
        first_name: co_parent_params[:first_name],
        last_name: co_parent_params[:last_name]
      )
      Parentage.create!(parent_person: co_parent, child_person: child_person, kind: co_parent_params[:relationship_to_child])
    when "deceased", "sole"
      # No co-parent Parentage row created. User has sole PR.
    end
  end

  def marriage_kind_from(relationship_kind, marriage_kind_param)
    return marriage_kind_param if marriage_kind_param.present?
    relationship_kind == "spouse" ? "married" : "cohabiting"
  end

  def destroy_existing_partnership!
    Marriage.where(will_maker_person: @will_maker).destroy_all
    @onboarding_session.partner_person&.destroy
  end
end
```


```ruby
class MigrateChildrenPayloadToRecords < ActiveRecord::Migration[8.1]
  def up
    OnboardingSession.where.not(children_payload: nil).find_each do |session|
      next if session.children_payload.blank?

      will_maker = session.will_maker_person
      next if will_maker.nil?  # session never reached welcome step; skip

      ActiveRecord::Base.transaction do
        Array(session.children_payload).each_with_index do |child_data, idx|
          child = session.people.create!(
            relationship_kind: "child",
            first_name: child_data["first_name"],
            last_name: child_data["last_name"],
            date_of_birth: child_data["date_of_birth"],
            disabled: child_data["disabled"] == "yes",
            lacks_mental_capacity: child_data["lacks_mental_capacity"] == "yes"
          )
          Parentage.create!(
            parent_person: will_maker,
            child_person: child,
            kind: child_data["relationship"] || "biological"
          )
        end
      end
    end
  end

  def down
    # Best-effort: reverse migration drops Person+Parentage rows for children.
    # If Epic 2 v1 ships and we need to rollback, run the down migration
    # before reintroducing the children_payload column. Lossy.
    OnboardingSession.find_each do |session|
      session.people.where(relationship_kind: "child").destroy_all
    end
  end
end
```

This runs **before** workstream 1.4 drops `children_payload`.

### 4.3 Stimulus controller updates — `app/javascript/controllers/family_form_controller.js`

Add new targets for the conditional reveals:

```javascript
import { Controller } from "@hotwired/stimulus"

export default class extends Controller {
  static targets = [
    "relationshipKind", "partnerFields",
    "hasChildren", "childrenSection", "childrenList",
    "coParentChoice", "partnerRelationship", "someoneElseFields"
  ]

  connect() {
    this.refreshConditionals()
  }

  onRelationshipChange() {
    this.refreshConditionals()
  }

  onHasChildrenChange() {
    this.refreshConditionals()
  }

  onCoParentChange(event) {
    const radio = event.target
    const card = radio.closest(".child-card")
    if (!card) return

    const partnerRel = card.querySelector("[data-co-parent-target='partnerRelationship']")
    const someoneElse = card.querySelector("[data-co-parent-target='someoneElseFields']")

    partnerRel.hidden = radio.value !== "partner"
    someoneElse.hidden = radio.value !== "someone_else"
  }

  refreshConditionals() {
    const relationshipKind = this.relationshipKindTarget.querySelector("input:checked")?.value
    const hasPartner = ["spouse", "civil_partnership", "cohabiting"].includes(relationshipKind)

    this.partnerFieldsTarget.hidden = !hasPartner
    this.element.querySelectorAll("[data-show-when-partnered]").forEach(el => {
      el.hidden = !hasPartner
    })

    const hasChildren = this.hasChildrenTarget.querySelector("input:checked")?.value === "yes"
    this.childrenSectionTarget.hidden = !hasChildren
  }

  addChild() {
    // Existing add-child logic — unchanged.
  }
}
```

### 4.4 Validations on the controller side

`Mobile::OnboardingController#family_params` enforces:

- if `relationship_kind` is `spouse|civil_partnership|cohabiting`: partner first/last name required, `started_at` required
- if `has_children` is `yes`: at least one child required, each child requires first_name + relationship_kind
- if any child has `co_parent.type = "partner"`: partner must exist (relationship_kind is partnered) AND partner_relationship_to_child required
- if any child has `co_parent.type = "someone_else"`: first_name + last_name + relationship_to_child required

Use Rails validation contexts (`:family_step`) on `Person` and `Parentage`.
